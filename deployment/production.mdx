## Overview

This guide covers production deployment strategies for VulnTrack, including database configuration, scaling, monitoring, and performance optimization.

  </Step>

  <Step title="Database Setup">

</Steps>

## Database Configuration

### Managed PostgreSQL (Recommended)

Use a managed database service for production:

<CodeGroup>
```bash AWS RDS
DATABASE_URL="postgresql://username:password@your-db.rds.amazonaws.com:5432/vulntrack?sslmode=require&connection_limit=20"
```

```bash Google Cloud SQL
DATABASE_URL="postgresql://username:password@/vulntrack?host=/cloudsql/project:region:instance&sslmode=require"
```

```bash Azure Database
DATABASE_URL="postgresql://username:password@your-db.postgres.database.azure.com:5432/vulntrack?sslmode=require"
```

```bash Supabase
DATABASE_URL="postgresql://postgres:password@db.project.supabase.co:5432/postgres?sslmode=require&pgbouncer=true"
```
</CodeGroup>

### Connection Pooling

For optimal performance, use connection pooling:

```typescript prisma/schema.prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
  directUrl = env("DIRECT_URL") // For migrations
}
```

```bash .env
# Pooled connection (for queries)
DATABASE_URL="postgresql://user:pass@host:5432/db?pgbouncer=true&connection_limit=10"

# Direct connection (for migrations)
DIRECT_URL="postgresql://user:pass@host:5432/db"
```

<Tip>
  Set `connection_limit` to match your application's concurrency needs. Start with 10-20 and adjust based on monitoring.
</Tip>

### Database Migrations

For production, use Prisma migrations instead of `db push`:

<Steps>
  <Step title="Generate migration">
    ```bash
    npx prisma migrate dev --name init
    ```
  </Step>

  <Step title="Review migration SQL">
    Check the generated SQL in `prisma/migrations/` directory.
  </Step>

  <Step title="Deploy to production">
    ```bash
    npx prisma migrate deploy
    ```
  </Step>
</Steps>

<Warning>
  Never use `prisma db push` in production as it may result in data loss. Always use migrations for schema changes.
</Warning>

### Backup Strategy

<AccordionGroup>
  <Accordion title="Automated Backups">
    Configure automated backups with your database provider:
    - **Frequency**: Daily full backups + continuous WAL archiving
    - **Retention**: 30 days for daily, 7 days for point-in-time
    - **Testing**: Regularly test restore procedures

    ```bash
    # Manual backup example
    pg_dump -h your-db.example.com -U vulntrack \
      -F c -b -v \
      -f backup_$(date +%Y%m%d).dump \
      vulntrack
    ```
  </Accordion>

  <Accordion title="Point-in-Time Recovery">
    Enable PITR for critical deployments:
    - Configure WAL archiving
    - Store archives in S3/GCS
    - Test recovery procedures monthly
  </Accordion>

  <Accordion title="Backup Verification">
    ```bash
    # Restore to test database
    pg_restore -h test-db.example.com -U vulntrack \
      -d vulntrack_test \
      --clean --if-exists \
      backup_20260303.dump
    
    # Verify data integrity
    psql -h test-db.example.com -U vulntrack -d vulntrack_test \
      -c "SELECT COUNT(*) FROM \"User\"; SELECT COUNT(*) FROM \"Vulnerability\";"
    ```
  </Accordion>
</AccordionGroup>

## Application Deployment

### Build Configuration

Update `package.json` build script for production:

```json package.json
{
  "scripts": {
    "build": "npx prisma generate && next build",
    "start": "next start",
    "migrate:deploy": "npx prisma migrate deploy"
  }
}
```

<Warning>
  Remove `prisma db push --accept-data-loss` from production builds. Use migrations instead.
</Warning>

## Scaling Strategies

### Horizontal Scaling

VulnTrack is stateless and can be horizontally scaled:

<Steps>
  <Step title="Load Balancer">
    Deploy multiple instances behind a load balancer:
    - Use health check endpoint: `/api/health`
    - Enable sticky sessions for WebSocket support (if added)
    - Configure SSL/TLS termination at load balancer
  </Step>

  <Step title="Database Read Replicas">
    For read-heavy workloads:
    ```typescript
    // Use read replica for queries
    const readReplica = new PrismaClient({
      datasources: {
        db: {
          url: process.env.DATABASE_READ_URL
        }
      }
    });
    ```
  </Step>

  <Step title="Caching">
    Implement caching for frequently accessed data:
    - Use Redis for session storage
    - Cache vulnerability statistics
    - Implement HTTP caching headers
  </Step>
</Steps>

    - Use Image optimization for logos/avatars
    - Implement route-level code splitting
    - Enable compression (gzip/brotli)
  </Accordion>

  <Accordion title="Monitoring Slow Queries">
    Enable Prisma query logging:
    ```typescript
    const prisma = new PrismaClient({
      log: [
        { level: 'query', emit: 'event' },
        { level: 'warn', emit: 'stdout' },
        { level: 'error', emit: 'stdout' }
      ]
    });
    
    prisma.$on('query', (e) => {
      if (e.duration > 1000) {
        console.warn('Slow query:', e.query, e.duration);
      }
    });
    ```
  </Accordion>
</AccordionGroup>

### Health Checks

Implement comprehensive health checks:

```typescript app/api/health/route.ts
import { prisma } from '@/lib/prisma';

export async function GET() {
  try {
    // Check database connection
    await prisma.$queryRaw`SELECT 1`;
    
    return Response.json({
      status: 'healthy',
      timestamp: new Date().toISOString(),
      database: 'connected'
    });
  } catch (error) {
    return Response.json(
      {
        status: 'unhealthy',
        timestamp: new Date().toISOString(),
        database: 'disconnected',
        error: error.message
      },
      { status: 503 }
    );
  }
}
```

## Zero-Downtime Deployments

For production environments, implement rolling updates:

```yaml kubernetes-deployment.yaml
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  minReadySeconds: 10
  template:
    spec:
      containers:
      - name: vulntrack
        livenessProbe:
          httpGet:
            path: /api/health
            port: 3000
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /api/health
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 5
```

## Troubleshooting

### Common Production Issues

<AccordionGroup>
  <Accordion title="High Database Connection Count">
    **Symptoms**: "Too many connections" errors
    
    **Solutions**:
    - Reduce `connection_limit` in DATABASE_URL
    - Implement connection pooling (PgBouncer)
    - Scale database instance
  </Accordion>

  <Accordion title="Memory Leaks">
    **Symptoms**: Increasing memory usage over time
    
    **Solutions**:
    - Use Node.js heap snapshots
    - Check for unclosed Prisma clients
    - Implement memory limits and auto-restart policies
  </Accordion>

  <Accordion title="Slow Query Performance">
    **Symptoms**: High response times, database CPU spikes
    
    **Solutions**:
    - Enable query logging
    - Add indexes to frequently queried fields
    - Use database query analyzer
  </Accordion>
</AccordionGroup>
